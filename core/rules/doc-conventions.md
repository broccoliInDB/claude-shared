# Doc Conventions

## 문서 상태 관리

### 상태 이모지

| 문서 | 상태 | 이모지 |
|------|------|--------|
| PRD/SPEC | 초안 | 📝 |
| PRD/SPEC | 검토중 | 🔍 |
| PRD/SPEC | 확정 | ✅ |
| PLAN 항목 | 미시작 | 🔴 |
| PLAN 항목 | 진행중 | 🟡 |
| PLAN 항목 | 완료 | 🟢 |

### 메타 정보 필수 필드

```markdown
> **작성일**: YYYY-MM-DD **문서 상태**: 📝 초안 **담당자**: {담당자}
> **규모**: S/M/L/XL **연결된 문서**: [SPEC](./SPEC.md) | [PLAN](./PLAN.md)
```

## 규모 기준

| 규모 | 기준 | plans/ 파일 수 |
|------|------|----------------|
| S | 단일 컴포넌트, API 1-2개 | 1개 |
| M | 화면 1개 + 컴포넌트 3-5개 | 2-3개 |
| L | 화면 2-3개 + 컴포넌트 5-10개 | 5-7개 |
| XL | 화면 4개+ 또는 복잡한 상태 | 8개+ |

## PLAN 정합성 규칙

PLAN.md 수정 시 **반드시** 3가지가 1:1 일치:
1. 대시보드 테이블 "상세 문서" 열
2. 📂 상세 문서 디렉토리 트리
3. 실제 `plans/` 파일

## 교차 참조 방향

```
PRD → SPEC 참조 (요구사항 → 기술 구현)
SPEC → PLAN 참조 (기술 구현 → 작업 계획)
PLAN → PRD/SPEC 역참조 (작업이 어떤 요구사항을 충족하는지)
```

## QA 체크리스트

문서 작성/수정 후:
- [ ] 메타 정보가 정확한가 (날짜, 상태, 담당자)
- [ ] 연결된 문서 링크가 유효한가
- [ ] PLAN 정합성 (대시보드 ↔ 트리 ↔ 파일) 일치하는가
- [ ] 소스 코드 참조가 실제 존재하는 파일인가
