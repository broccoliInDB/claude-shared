# Multi-Agent

## 작업 분배 원칙

```
대규모 작업 감지
       ↓
서브태스크 문서 작성
       ↓
병렬 가능 작업 식별
       ↓
에이전트 병렬 실행
       ↓
모니터링 & 보고
       ↓
검증 & QA
```

### 병렬화 기준

- **독립적 도메인**: Backend / Frontend / QA는 병렬 가능
- **의존성 확인**: Phase 간 의존성이 있으면 순차 실행
- **리소스 충돌**: 같은 파일 수정 시 순차 실행

## 서브태스크 문서화

병렬 작업 시작 전 반드시 서브태스크 문서 작성:

```markdown
# 서브태스크 분배

## 팀 구성
- Agent A: [담당 범위]
- Agent B: [담당 범위]

## 의존성
- Phase X는 Phase Y 완료 후 진행

## 완료 체크리스트
- [ ] Agent A 완료
- [ ] Agent B 완료
- [ ] 통합 검증 통과
```

## 에스컬레이션

다음 상황에서는 **즉시 사용자에게 보고하고 의사결정 요청**:

| 상황 | 보고 내용 |
|------|----------|
| **설계 결정** | 여러 접근법 중 선택 필요 |
| **영향도 큰 변경** | 기존 데이터/API 호환성 |
| **충돌 발생** | 에이전트 간 작업 충돌 |
| **예상치 못한 복잡도** | 작업량 증가 |
| **에러/블로커** | 진행 불가 상황 |

## Context Isolation Protocol

- 한 번에 **하나의 기능**과 **하나의 단계**에만 집중
- 대화 시작 시 기능이 지정되지 않았다면 먼저 질문
- 기능별 spec/plan/context 파일이 있으면 해당 폴더만 진실의 원천으로 취급

### context.md 관리

세션 간 맥락 유지가 필요한 장기 작업 시:

```markdown
# Context - {feature-name}

## 완료된 작업
- ...

## 현재 진행 중
- ...

## 발생한 이슈/결정 사항
- ...

## 다음 세션에서 이어갈 작업
- ...
```

업데이트 시점:
- 코드 변경 시마다 최신 상태 유지
- Context Window 부족 시 상태 요약 후 세션 초기화 요청

## 완료 후 정리

모든 서브태스크 완료 후:
1. 서브태스크 체크리스트 업데이트
2. 전체 변경 사항 요약 (사용자에게 보고)
3. 통합 테스트 실행
4. context.md 업데이트
